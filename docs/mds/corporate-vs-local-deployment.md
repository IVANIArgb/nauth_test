# Корпоративный ПК и домашняя разработка: почему ведёт себя по‑разному

Один репозиторий не гарантирует одинаковое поведение: отличаются **окружение**, **политики домена** и **файлы вне git** (`.env`, база, каталог контента).

## Что почти всегда разное

| Фактор | Домашний dev | Корпоративный ПК |
|--------|----------------|-------------------|
| `.env` | Ваши пути, часто `TEST_MODE`, локальный `CONTENT_ROOT_DIR` | Другой пользователь Windows, другой диск, запрет на чужие `C:\Users\...` |
| Kerberos / Negotiate | Часто нет билета на `localhost` | Есть домен, SPN, политики браузера (intranet, AuthServerAllowlist) |
| Обратный прокси | Нет | IIS/nginx с `Remote-User` / `TRUST_REMOTE_USER` |
| `FLASK_ENV=production` | Редко | Часто: жёсткие проверки в `backend/__init__.py` |
| CSP / антивирус | Мягче | Режут CDN, внешние шрифты, иногда локальные порты |

## Что уже сделано в коде

1. **Недоступный `CONTENT_ROOT_DIR`** — откат на `categories-data` в корне проекта + при старте `run.py` сброс неверной переменной в `os.environ`, если путь не читается текущим процессом.
2. **`is_external_content_root()`** — считает «внешний» корень только если **фактический** путь из `get_base_categories_data_path()` не совпадает с встроенным `categories-data` (после отката не ломается логика дубликатов).
3. **При старте приложения** в лог пишется строка `learningsite.startup`: `FLASK_ENV`, платформа, Docker, **реальный путь контента**, `TEST_MODE`, `TRUST_REMOTE_USER`, `KERBEROS_GSSAPI_ENABLED` — сравните две машины по этой строке.

## Чеклист при расхождении «дома работает — на работе нет»

1. Откройте на обоих ПК лог сервера и сравните строку `Старт: FLASK_ENV=...`.
2. Сравните **только** `env.example` → свой `.env`: не копируйте чужой `.env` целиком; для корпоративного ПК заново пропишите `CONTENT_ROOT_DIR` под **того пользователя**, от которого запускается Python/служба.
3. Убедитесь, что на корпоративном не включён случайно `FLASK_ENV=testing` (иначе отключается часть сценариев Windows-идентичности в auth).
4. Kerberos в браузере: для не-`localhost` URL добавьте сайт в intranet / списки SSO (см. `docs/mds/kerberos-ad-flask-docker.md`).
5. Если контент за прокси с SSO: `TRUST_REMOTE_USER=true` + корректные `REMOTE_USER_HEADERS` + в production `TRUST_REMOTE_USER_CONFIRM=true`.

## Реалистичные ожидания

«100% одинаково на всех машинах» без унификации запуска (Docker Compose, одинаковый `.env` по шаблону, одна версия Python) недостижимо. Цель — **предсказуемый откат**, **явные логи** и **чеклист**, а не магическая одинаковость ОС и AD.
